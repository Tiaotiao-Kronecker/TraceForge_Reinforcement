# TraceForge 推理输出数据结构说明（原始格式）

本文描述**未改动的** TraceForge `infer.py` 所写出的推理结果，便于对已有 10000+ 轨迹做 3D 可视化。

---

## 1. 输出目录布局

一次推理（单路视频）后，输出目录大致为：

```
{out_dir}/
└── {video_name}/                    # 如 images0
    ├── {video_name}.npz             # 即 images0.npz，视频级数据（见下）
    ├── images/                      # 每个查询帧一张 RGB 图（该帧截图）
    │   ├── images0_0.png
    │   ├── images0_5.png
    │   └── ...
    ├── depth/                       # 每个查询帧一张深度图
    │   ├── images0_0.png
    │   └── ...
    └── samples/                     # 每个查询帧一段轨迹，一个 NPZ
        ├── images0_0.npz
        ├── images0_5.npz
        └── ...
```

- **查询帧**：由 `frame_drop_rate` 决定（如 0, 5, 10, …），即从第 0 帧起每隔若干帧取一帧作为「查询帧」。
- **每个查询帧**：从该帧的 20×20 网格 keypoint 出发，向前跟踪**固定时间窗口 T**（如 16 或 128，由 `future_len` 控制），得到 **一段** 轨迹。这段轨迹在时间上对应 **该查询帧起的 T 帧**。
- 不同查询帧的轨迹，可以理解为**同一段物理运动在不同起始时刻的采样**；每一段都可以单独做「从该段第 0 帧到第 T 帧」的 3D 动画。

---

## 2. images0.npz（视频级 NPZ）

- **路径**：`{out_dir}/{video_name}/{video_name}.npz`（例如 `images0/images0.npz`）。

| 键 | 形状 | 说明 |
|----|------|------|
| `coords` | `(T1, N, 3)` | **仅第一段**的 3D 轨迹（世界坐标），T1 = 第一段长度（≤ future_len），N=400。 |
| `depths` | `(T1, H, W)` | **仅第一段**的深度图，与 coords 对齐。整段视频的深度未写入。 |
| `intrinsics` | `(T_video, 3, 3)` | 整段视频每帧内参。 |
| `extrinsics` | `(T_video, 4, 4)` | 整段视频每帧外参（w2c）。可视化时需 `c2w = np.linalg.inv(extrinsics[t])`。 |
| `visibs` | `(T1, N, 1)` | 第一段轨迹可见性。 |
| `height`, `width` | 标量 | 图像高宽。 |
| `unc_metric` | `(T1, H, W)` | 第一段深度不确定性。 |
| `video` | `(T1, H, W, 3)` 可选 | 若 `--save_video`，为**第一段** RGB，非整段视频。 |

要点：**depths / video / coords 只覆盖第一段**；其它段的深度和视频未写入此 NPZ，因此对非第一段无法从磁盘恢复「每像素点云」，只能做 **keypoint 轨迹** 的 3D 动画。

---

## 3. samples/images0_{f}.npz（按查询帧的轨迹 NPZ）

- **路径**：`{out_dir}/{video_name}/samples/images0_{f}.npz`，`f` 为查询帧下标（0, 5, 10, …）。
- **含义**：从第 `f` 帧起、长度为 **T** 的一段轨迹，与「该段」内 T 帧一一对应。

| 键 | 形状 | 说明 |
|----|------|------|
| `frame_index` | `(1,)` 或标量 | 查询帧下标 `f`。 |
| `keypoints` | `(N, 2)` | 查询帧上的 2D 关键点（像素坐标），N=400（20×20）。 |
| `traj` | `(N, T, 3)` | 该段每条轨迹的 3D 位置。**原始 infer 中**由 `project_tracks_3d_to_3d` 写入：为**首帧相机下的像平面 + 深度**空间，即 (x_img, y_img, z)，x∈[0,W]，y∈[0,H]，z 为深度相关量，非世界坐标。 |
| `valid_steps` | `(N, T)` | 每点每步是否有效。 |
| `traj_2d` | `(N, T, 2)` | 2D 投影轨迹（首帧相机），用于 2D 可视化。 |
| `image_path` | 等 | 该查询帧对应图像路径（如 `images/images0_{f}.png`）。 |

要点：**每个 sample NPZ 独立描述一段「从该段第 0 帧到第 T 帧」的 keypoint 轨迹**；不需要整段视频的深度，即可做这一段 keypoint-only 的 3D 动画（轨迹在相机/像空间仍可视为 3D 点与线）。

---

## 4. 与「每查询帧一段 3D 动画」的对应关系

- 每个 **samples/images0_{f}.npz** 对应**一个查询帧 f** 和**一段固定长度 T**。
- 要做类似 [SpaTrackerV2](https://github.com/henry123-boy/SpaTrackerV2) 的 3D 动画，可以**按查询帧分别做**：
  - **第一段（f=0）**：可用 images0.npz 的 `coords`、`depths`、相机、可选 `video`，做「keypoint + 可选稠密点云」的 3D 动画。
  - **其它段（f>0）**：磁盘上只有该段的 sample NPZ，没有该段的深度/视频，因此**只做 keypoint 轨迹的 3D 动画**即可满足需求（每 keypoint 从段内第 0 帧到第 T 帧的运动）。
- 不要求「每一像素」的 3D 动画；**每个 keypoint 的 3D 动画**即足够。

---

## 5. 可视化时的数据使用方式

- **仅用 sample NPZ**：读取 `traj` (N, T, 3)、`frame_index`、`valid_steps`。将 `traj` 视为 3D 点（相机/像空间），按时间步 0…T-1 绘制点与连线，即可得到该段 keypoint-only 的 3D 动画。
- **第一段且需点云时**：再读 images0.npz，用其 `depths`、`intrinsics`、`extrinsics` 做反投影得到每帧点云，与 `coords` 轨迹一起显示。
